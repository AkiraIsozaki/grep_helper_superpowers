# プロダクト要求定義書 (Product Requirements Document)

## プロダクト概要

### 名称
**grep_analyzer** - grep結果 自動分類・使用箇所洗い出しツール（Java / C / Pro*C / SQL / Shell / Kotlin / PL/SQL / TypeScript・JS / Python / Perl / C#・VB.NET / Groovy 対応）

### プロダクトコンセプト
- **網羅的な使用箇所洗い出し**: grepでヒットした文言の直接参照だけでなく、定数/変数/getter経由の間接参照まで追跡し、1件の見落としもなく全件洗い出す（Java: 3段階、C/Pro*C/Kotlin: 直接参照 + #define/const定数経由の間接参照、C#/VB.NET: 直接参照 + Const/static readonly定数経由の間接参照、Groovy: 直接参照 + static final定数・フィールド経由の間接参照 + setter追跡、SQL/Shell/PL/SQL/TypeScript・JS/Python/Perl: 直接参照のみ）
- **使われ方の自動分類**: 言語ごとの使用パターンを自動判定し、人間の目視確認コストを大幅に削減する（Javaは javalang AST解析 + 正規表現フォールバック、その他はすべて正規表現分類）
- **どこでも即使える配布形式**: Python venv + zip配布により、解凍してすぐ使い始められる。実行環境の構築作業が不要

### プロダクトビジョン
Java・C・Pro*C・SQL・Shell・Kotlin・PL/SQLスクリプト混在プロジェクト内の特定の文言（文字列値・コード値）を転用・削除しようとする開発者が、「この値は今どこでどのように使われているか」を完全に把握できるようにする。grepだけでは拾えない定数経由・getter経由の間接的な使用箇所まで追跡し、TSV形式で一覧化することで、数万行の手作業確認を不要にする。

### 目的
- grep結果4万行規模の手作業目視確認を自動化する
- 言語ごとの使われ方パターンに自動分類し、影響範囲の把握を容易にする
- 直接参照だけでなく間接参照（定数/変数/getter経由）まで追跡し、転用・削除時の取りこぼしをゼロにする
- Excelで開いてフィルタ・ソートできるTSVを出力し、チームでの調査共有を容易にする

---

## ターゲットユーザー

### プライマリーペルソナ: Java・C・Pro*C混在レガシーシステムのリファクタリング担当者

- 大規模プロジェクト（数千〜数万ファイル、Java/C/Pro*C/SQL/Shell混在）を保守・改修している
- 特定の文字列値・コード値（ビジネスコード・ステータス値など）の意味を変更・廃止したい
- その文字列が現在どこでどのように使われているかを全件調査する必要がある
- **現在の課題**: grepで4万行ヒットしても、1行ずつ確認するのは現実的でない。また定数経由の間接使用はgrepでは拾えず、見落としが発生するリスクがある
- **期待する解決策**: 全件を自動分類したTSVを受け取り、Excelでフィルタしながら方針を決めたい
- **典型的なワークフロー**: grepを手動実行 → input/に配置 → 言語別ツール実行 → output/のTSVをExcelで確認

## 成功指標(KPI)

### プライマリーKPI
- **網羅率**: 既知のテストケース（直接参照・間接参照・getter経由の各パターンを含むサンプルJavaファイル群）を用意し、期待出力と一致すること（全件出力・0件見落とし）
  - 評価方法: テスト用Javaソース一式と期待TSVを事前定義し、実行結果と比較する
  - 合格基準: 期待TSVの全行が出力に含まれること
- **分類精度**: 7種の使用タイプへの自動分類が正しく行われること
  - 評価方法: テスト用Javaソース50ファイル（各使用タイプ10件以上）を用意し、手動ラベルと自動分類結果を比較
  - 合格基準: 一致率90%以上（450/500件以上が一致）
  - 評価実施タイミング: リリース前に必ず実施

### セカンダリーKPI
- **セットアップ時間**: zip解凍〜初回実行まで5分以内（配布zipに同梱するREADME.txtの手順に従って）
- **処理速度**: 4万行・500ファイル規模を30分以内を目安とする（ただし時間制限は設けない、網羅性優先）

## 機能要件

### コア機能(MVP)

#### F-01: grep結果の読み込みとパース

**ユーザーストーリー**:
調査担当者として、手動実行したgrepの結果ファイルをツールに渡すだけで分析を開始したいため、input/ディレクトリにファイルを置くだけで自動処理してほしい

**受け入れ条件**:
- [ ] `input/` ディレクトリ内の `.grep` ファイルを全て自動検出する
- [ ] ファイル名（拡張子なし）を「文言」列の値として使用する
- [ ] `ファイルパス:行番号:コード行` 形式のgrep行を正しくパースする
- [ ] バイナリファイル通知行・空行・不正フォーマット行はスキップし、件数をレポートに記録する
- [ ] `input/` が空の場合はエラーメッセージを標準エラー出力（stderr）に日本語で表示し、exit code 1 で終了する
- [ ] 共通インフラ（`parse_grep_line`, `write_tsv`, `GrepRecord`, `ProcessStats`）は `analyze_common.py` に集約し、全言語アナライザーから共有する

**優先度**: P0（必須）

---

#### F-02: 第1段階 — 直接参照の分類

**ユーザーストーリー**:
調査担当者として、grepでヒットした各行が「定数定義なのか」「条件判定なのか」を知りたいため、コードの使用パターンを言語別に自動分類してほしい

**受け入れ条件（Java: `analyze.py`）**:
- [ ] `javalang` によるAST解析で以下の7種を分類する: アノテーション・定数定義・変数代入・条件判定・return文・メソッド引数・その他
- [ ] コメント行もスキップせず「その他」として出力する
- [ ] `javalang` がパースエラーを起こした場合は正規表現フォールバックで継続し、レポートに記録する

**受け入れ条件（C: `analyze_c.py`）**:
- [ ] 正規表現で以下の6種を分類する: #define定数定義・条件判定・return文・変数代入・関数引数・その他

**受け入れ条件（Pro*C: `analyze_proc.py`）**:
- [ ] ファイル拡張子（`.pc` / `.c` / `.h`）に基づいて分類関数を切り替える（`.c`/`.h` は `classify_usage_c` を使用）
- [ ] 正規表現で以下の7種を分類する: EXEC SQL文・#define定数定義・条件判定・return文・変数代入・関数引数・その他

**受け入れ条件（SQL: `analyze_sql.py`）**:
- [ ] 正規表現で以下の7種を分類する: 例外・エラー処理・定数・変数定義・WHERE条件・比較・DECODE・INSERT/UPDATE値・SELECT/INTO・その他

**受け入れ条件（Shell: `analyze_sh.py`）**:
- [ ] 正規表現で以下の6種を分類する: 環境変数エクスポート・変数代入・条件判定・echo/print出力・コマンド引数・その他

**受け入れ条件（Kotlin: `analyze_kotlin.py`）**:
- [ ] 正規表現で以下の7種を分類する: const定数定義・変数代入・条件判定・return文・アノテーション・関数引数・その他
- [ ] `const val` 定義行から定数名を抽出し、`.kt`/`.kts` ファイルを対象にプロジェクト全体を追跡する（間接参照）

**受け入れ条件（PL/SQL: `analyze_plsql.py`）**:
- [ ] 正規表現で以下の7種を分類する: 定数/変数宣言・EXCEPTION処理・条件判定・カーソル定義・INSERT/UPDATE値・WHERE条件・その他
- [ ] 直接参照のみ（間接追跡なし）

**受け入れ条件（TypeScript/JavaScript: `analyze_ts.py`）**:
- [ ] 正規表現で以下の7種を分類する: const定数定義・変数代入(let/var)・条件判定・return文・デコレータ・関数引数・その他
- [ ] 直接参照のみ（間接追跡なし）

**受け入れ条件（Python: `analyze_python.py`）**:
- [ ] 正規表現で以下の6種を分類する: 変数代入・条件判定・return文・デコレータ・関数引数・その他
- [ ] 直接参照のみ（間接追跡なし）

**受け入れ条件（Perl: `analyze_perl.py`）**:
- [ ] 正規表現で以下の6種を分類する: use constant定義・変数代入・条件判定・print/say出力・関数引数・その他
- [ ] 直接参照のみ（間接追跡なし）

**受け入れ条件（C#/VB.NET: `analyze_dotnet.py`）**:
- [ ] 正規表現で以下の7種を分類する: 定数定義(Const/readonly)・変数代入・条件判定・return文・属性(Attribute)・メソッド引数・その他
- [ ] `const` / `static readonly` 定義行から定数名を抽出し、`.cs`/`.vb` ファイルを対象にプロジェクト全体を追跡する（間接参照）

**受け入れ条件（Groovy: `analyze_groovy.py`）**:
- [ ] 正規表現で以下の7種を分類する: static final定数定義・変数代入・条件判定・return文・アノテーション・メソッド引数・その他
- [ ] `static final` 定数・フィールドのプロジェクト全体追跡（間接参照）
- [ ] setter経由の代入箇所追跡（setter追跡）

**共通受け入れ条件**:
- [ ] 参照種別 = `直接` として出力する

**優先度**: P0（必須）

---

#### F-03: 第2段階 — 変数/定数経由の間接参照追跡

**ユーザーストーリー**:
調査担当者として、「この定数に値を代入しているのはわかったが、その定数はどこで使われているか」まで知りたいため、間接的な使用箇所も自動追跡してほしい

**受け入れ条件**:
- [ ] 第1段階で「定数定義」に分類された行から変数名を抽出し、プロジェクト全体を追跡する
- [ ] 第1段階で「変数代入（フィールド）」に分類された行は同一クラス内を追跡する
- [ ] 第1段階で「変数代入（ローカル変数）」に分類された行は同一メソッド内を追跡する
- [ ] 各出現箇所をAST解析して7種の使用タイプに分類する
- [ ] 参照種別 = `間接` として出力する
- [ ] 参照元変数名・参照元ファイル・参照元行番号を記録する
- [ ] 定義行自体は第1段階で記録済みのため重複追加しない

**優先度**: P0（必須）

---

#### F-04: 第3段階 — getter経由の間接参照追跡

**ユーザーストーリー**:
調査担当者として、privateフィールドがgetter経由で外部に渡っている場合も把握したいため、getter呼び出し箇所もプロジェクト全体で追跡してほしい

**受け入れ条件**:
- [ ] フィールドの同一クラス内から、命名規則（`type` → `getType()`）でgetter候補を特定する
- [ ] `return フィールド名;` しているメソッドも非標準命名として拾う
- [ ] 特定したgetter名でプロジェクト全体を検索し、呼び出し箇所をAST解析する
- [ ] 参照種別 = `間接（getter経由）` として出力する
- [ ] 他クラスの同名getterによるfalse positiveは許容し、全件出力する（もれなく優先）

**優先度**: P0（必須）

---

#### F-05: TSV出力

**ユーザーストーリー**:
調査担当者として、分析結果をExcelで開いてフィルタ・確認したいため、UTF-8 BOM付きのTSVで出力してほしい

**受け入れ条件**:
- [ ] 出力列: 文言・参照種別・使用タイプ・ファイルパス・行番号・コード行・参照元変数名・参照元ファイル・参照元行番号
- [ ] 参照種別の取りうる値: `直接` / `間接` / `間接（getter経由）`
- [ ] エンコード: UTF-8 BOM付き（Excelで文字化けなし）
- [ ] ソート順: 文言 → 直接参照の(ファイルパス → 行番号) → 直接参照が先・間接参照が後（直接参照の直後にその間接参照が続くグループ順）
- [ ] 入力ファイルごとに対応するTSVを `output/` に出力する（例: `input/TARGET.grep` → `output/TARGET.tsv`）
- [ ] `output/` ディレクトリが存在しない場合は自動作成する
- [ ] 出力例:
  ```
  文言	参照種別	使用タイプ	ファイルパス	行番号	コード行	参照元変数名	参照元ファイル	参照元行番号
  TARGET	直接	定数定義	Constants.java	10	public static final String CODE = "TARGET"
  TARGET	間接	条件判定	Service.java	110	if (someVar.equals(CODE)) {	CODE	Constants.java	10
  TARGET	間接（getter経由）	メソッド引数	Handler.java	55	someService.process(obj.getCode());	getCode	Entity.java	8
  TARGET	直接	条件判定	Validator.java	80	if (x.equals("TARGET")) {
  ```

**優先度**: P0（必須）

---

#### F-06: 処理レポートの出力

**ユーザーストーリー**:
調査担当者として、処理中にスキップされた行やフォールバックが発生した箇所を把握したいため、処理完了後にレポートを表示してほしい

**受け入れ条件**:
- [ ] 標準出力に処理サマリを出力する（総行数・有効行数・スキップ行数）
- [ ] スキップした行の件数と代表的な内容を記録する
- [ ] javalangパースエラーでフォールバックしたファイル名・行番号を記録する
- [ ] エンコーディングエラーが発生したファイルを記録する

**優先度**: P0（必須）

### CLIインターフェース

言語ごとに専用スクリプトを使用する:

| 言語 | エントリーポイント | 説明 |
|------|-----------------|------|
| Java | `python analyze.py` | Java 3段階分析（AST + 間接参照 + getter追跡） |
| C | `python analyze_c.py` | 純C grep結果分類 |
| Pro*C | `python analyze_proc.py` | Pro*C（`.pc`/`.c`混在）grep結果分類 |
| Oracle SQL | `python analyze_sql.py` | SQL grep結果分類 |
| Shell | `python analyze_sh.py` | Bash/CSH/TCSH grep結果分類 |
| Kotlin | `python analyze_kotlin.py` | Kotlin grep結果分類（const val間接追跡あり） |
| PL/SQL | `python analyze_plsql.py` | PL/SQL grep結果分類（直接参照のみ） |
| TypeScript/JavaScript | `python analyze_ts.py` | TypeScript/JS grep結果分類（直接参照のみ） |
| Python | `python analyze_python.py` | Python grep結果分類（直接参照のみ） |
| Perl | `python analyze_perl.py` | Perl grep結果分類（直接参照のみ） |
| C#/VB.NET | `python analyze_dotnet.py` | C#/VB.NET grep結果分類（const/static readonly間接追跡あり） |
| Groovy | `python analyze_groovy.py` | Groovy grep結果分類（static final間接追跡・setter追跡あり） |

```bash
# Java（基本）
python analyze.py --source-dir /path/to/javaproject

# Pro*C
python analyze_proc.py --source-dir /path/to/proc/src

# オプション指定（全言語共通）
python analyze_proc.py --source-dir /path/to/src \
       --input-dir /custom/input \
       --output-dir /custom/output
```

| オプション | 必須 | デフォルト | 説明 |
|---|---|---|---|
| `--source-dir` | ✅ | - | 解析対象ソースコードのルートディレクトリ |
| `--input-dir` | - | `input/` | grep結果ファイルの配置ディレクトリ |
| `--output-dir` | - | `output/` | TSV出力先ディレクトリ |
| `--encoding` | - | 自動検出 | 文字コード強制指定（Kotlin/PL/SQL等の新言語アナライザーで利用可） |
| `--help` | - | - | 使い方を表示して終了 |

**終了コード**:

| コード | 意味 |
|---|---|
| `0` | 正常終了（TSV出力完了） |
| `1` | 入力エラー（`input/` が空・`--source-dir` 未指定など） |
| `2` | 実行時エラー（予期しない例外） |

## 非機能要件

### 網羅性（最優先）
- 直接参照・間接参照を問わず、1件の見落としもないこと
- 分類できないものは「その他」として必ず出力する（スキップしない）
- 処理時間を犠牲にしても網羅性を優先する

### パフォーマンス
- 処理時間の上限は設けない（数分かかっても許容）
- ASTキャッシュにより同一ファイルの再解析を省略する（ただし網羅性を損なわない範囲で）

### ユーザビリティ
- zip解凍〜初回実行まで5分以内（配布zipに同梱するREADME.txtの手順に従って）
- Windows / Mac / Linux で動作すること
- `--help` オプションで使い方を確認できること
- エラーメッセージは日本語で標準エラー出力（stderr）へ出力する

### 信頼性
- javalangパースエラー発生時も処理を中断せず継続する（フォールバック）
- エンコーディングエラー発生時も `errors='replace'` で継続する
- 処理完了後に必ずレポートを出力し、スキップ・フォールバックの件数を通知する

### 入出力
- **入力文字コード**: Shift-JIS（Javaソースファイル）
- **出力文字コード**: UTF-8 BOM付き（TSVファイル）
- **grep結果形式**: `ファイルパス:行番号:コード行`（標準grep -rn 形式）

### 配布
- Python 3.12以上 + venv で動作する
- 外部依存は `javalang` 1パッケージのみ
- zip配布、解凍してすぐ使える構成

## スコープ外

- IDE連携・GUIの提供
- データフロー解析（taint analysis）による完全なデータ伝播追跡
- setter経由の逆伝播追跡
- 未対応の言語のソース解析
- データベースへの結果保存
- CI/CDへの組み込み
- 複数文字コードの自動検出（MVP時点）
- 並列処理・マルチスレッド化（シングルプロセス設計を維持）
